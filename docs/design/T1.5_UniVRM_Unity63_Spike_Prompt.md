# T1.5 Agent Prompt — UniVRM × Unity 6.3 LTS Compatibility Spike

Copy the block below and paste it as the initial prompt to your coding agent (Claude Code, Cursor, etc.). The agent should have: a machine with Unity Hub installed, network access, an Android device in developer mode connected via USB, and read access to the project's `AI_Avatar_App_Design_Doc.md`.

---

## Prompt

You are implementing **Task T1.5** from `AI_Avatar_App_Design_Doc.md` (Section 10.1): a throwaway compatibility spike that proves UniVRM 0.131.x works on Unity 6.3 LTS on a physical Android device. This is a **gating task** — downstream VRM work cannot begin until this passes or its failures have documented workarounds.

### Setup

1. Install **Unity 6.3 LTS**, latest patch (version string `6000.3.x` — pick the newest `.x` available in Unity Hub). Include the Android Build Support module with OpenJDK, Android SDK & NDK Tools, and IL2CPP.
2. Create a new Unity project at `spikes/univrm-unity63/` using the 3D (URP) template. Do not add this folder to the main app's assembly.
3. Install **UniVRM 0.131.x** via the Package Manager using the VRM Consortium's official instructions for UPM installation (git URL or .unitypackage — follow whichever the release notes at the current 0.131.x release recommend).
4. Drop a VRM 1.0 test avatar into `Assets/TestAssets/`. If none is available, download the official VRM sample `AliciaSolid.vrm` (search: "AliciaSolid VRM sample") or use any creator-licensed VRM 1.0 file.

### Checks to execute

Run each check in order. For each, record in `spikes/univrm-unity63/RESULT.md` one of: **PASS**, **PASS-WITH-WORKAROUND** (describe the workaround), or **FAIL** (describe the failure mode, link the UniVRM issue if one exists).

**Check 1 — Runtime VRM load.** Write a MonoBehaviour that, on Start, uses `Vrm10.LoadPathAsync` (or the current equivalent in 0.131.x) to load the test VRM from `Application.streamingAssetsPath`. Verify the avatar renders in the scene. Do not use editor-time import for this check — it must be a runtime load.

**Check 2 — Blendshapes.** Extend the MonoBehaviour with three UI buttons: "Joy", "Angry", "Viseme A". Each button sets the corresponding VRM10 expression weight to 1.0 via the runtime expression API. Verify the avatar's face visibly changes. Reset to 0 after 1 second.

**Check 3 — Humanoid animation.** Add a humanoid `AnimationClip` (a simple Mixamo wave or idle retargeted to the VRM rig is fine) on an override layer. Play it via an Animator Controller. Verify the avatar moves cleanly — no broken bone chains, no foot slide exceeding ~5cm, no T-pose flashes. Specifically confirm the behavior from UniVRM Issue #2646 (animator controllers becoming broken) is NOT present on this Unity patch version.

**Check 4 — Android build.** Switch platform to Android. Configure: ARM64 target architecture, IL2CPP scripting backend, minimum API 26, target API 34+. Build an APK. Install and launch on a physical Android device (Android 12 or newer). Verify the scene loads and the VRM renders on device.

**Check 5 — SpringBones on device.** On the device build, either play the humanoid animation or briefly rotate the root transform in code. Visually confirm that hair, cloth, or accessory SpringBones animate secondarily. Record a short screen capture if possible.

### Deliverables

Commit to the branch `spike/univrm-unity63`:
- The Unity project at `spikes/univrm-unity63/` (include `.gitignore` for Library, Temp, obj, Build).
- `spikes/univrm-unity63/RESULT.md` with one section per check, the verdict, and any workaround code snippets.
- A top-level summary at the end of `RESULT.md` stating: (a) whether OQ-1 can be closed as confirmed, (b) any caveats downstream tasks must know about, (c) the exact Unity patch version and UniVRM version tested.
- If any check is **FAIL** without a workaround, do **not** merge the branch. Open a discussion thread summarizing the failure and proposed fallback (Unity 6.0 LTS migration plan, or alternative VRM loader evaluation).

### Non-goals for this spike

Do not wire up LLM providers, TTS, STT, or production UI. Do not try to optimize startup time. Do not polish the visuals. This project is disposable code; the `RESULT.md` is the artifact that matters.

### Questions you may need to answer by doing

- The exact UniVRM installation method (UPM git URL vs .unitypackage) changes occasionally; check the 0.131.x release notes rather than assuming.
- If Check 3 fails due to Issue #2646, try downgrading UniVRM one patch (0.130.x) or upgrading Unity to the latest 6.3 patch — record which combination works.
- If Check 4 fails at IL2CPP link time, check the UniVRM issue tracker for any Unity 6-specific `link.xml` requirements and try disabling code stripping as a diagnostic.

Exit when `RESULT.md` is complete and committed. Timebox: 2 working days. If you are still blocked at that point, escalate with the last known state rather than pushing further.
